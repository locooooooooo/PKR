# PKR v0.5 Reference Runtime

- Status: Implemented milestone
- Index: [PKR specification index](README.md)
- Scope: Local transactional Runtime, deterministic projections, and JSON CLI
- Depends on: [PKR v0.4 Coordination Record and Action Catalog](0010-pkr-v0.4-record-action-catalog.md)
- Successor: [PKR v0.6 Steward and LPS Orchestration](0012-pkr-v0.6-steward-lps-orchestration.md)
- Publication: Not independently released

## 1. Purpose

v0.5 turns the v0.2 and v0.4 contracts into the smallest restart-safe local
Runtime. It establishes one transactional authority for governed project state
and exposes that authority through a machine-readable command line.

## 2. Runtime contract

The reference implementation MUST:

- keep authoritative metadata, records, commands, and ordered events in one
  SQLite store;
- process each mutation atomically and persist its idempotent result;
- reject stale expected revisions and changed reuse of an idempotency key;
- rebuild deterministic JSON projections from authority;
- support bootstrap, Goal, Task, Agent, dispatch, callback, Verification,
  Workspace, event, status, and projection operations;
- return structured JSON results and errors from the CLI.

The storage layout follows the
[reference Runtime decision](../docs/decisions/0001-reference-runtime-layout.md).
SQLite is the reference implementation choice, not a requirement imposed on
every conforming PKR implementation.

## 3. Authority and completion

Projection files MUST NOT be accepted as mutation input. A Task reaches its
terminal success state only after the required Artifact and independent
Verification records exist and the Runtime accepts the governed transition.
A callback or participant report alone cannot complete work.

## 4. Recovery invariants

- Restart MUST reconstruct the same committed sequence and state digest.
- A failed transaction MUST expose no partial records or events.
- Replaying an identical command MUST return the original result without new
  events.
- Projection tampering MUST be repaired from SQLite authority.
- One CLI invocation MUST NOT report success before its transaction commits.

## 5. Compatibility role

v0.5 is an implemented and verified development milestone, not a standalone
public release or a stable compatibility promise. Its authority, idempotency,
revision, event-ordering, and projection rules are carried into the
[v1 stable contract](../docs/release/v1-stable-contract.md).

## 6. Non-goals

v0.5 does not claim a hosted service, distributed consensus, production Agent
integration, package registry, OS sandbox, or autonomous evolution.
