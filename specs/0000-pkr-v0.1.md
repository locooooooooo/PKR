# PKR v0.1 Definition

- Status: Vision baseline
- Phase: 0 - Specification
- Short name: PKR
- Architectural name: Project Kernel Runtime
- Initial public name: Project Knowledge Runtime

## 1. Vision

PKR is an open runtime for AI-native software development.

It does not exist to help an AI write code. It exists so humans, agents,
tools, and workflows follow the same project operating rules. A developer or
agent joining a project should not need to reconstruct the project from files
and conversation history. After loading PKR, it should immediately operate as
a member of the project organization.

## 2. Positioning

A conventional project centers the repository. PKR centers the project
runtime. The repository is no longer the project control plane; it is one
artifact source managed by the project runtime.

A project has one authoritative information state. Humans, Agents across
different model families, Agent hosts, development tools, GitHub, CI, and
monitoring consume PKR instead of independently interpreting a README.

PKR does not own code generation. PKR owns the rules by which a project keeps
running.

## 3. Runtime modules

PKR v0.1 defines eight logical runtime modules. A module is a responsibility,
not necessarily a process, service, directory, or persisted object kind.

1. **Mission Runtime** defines why the project exists, its target outcome,
   success criteria, budget, and deadline.
2. **Knowledge Runtime** provides authoritative structured facts, capabilities,
   prompts, documents, decisions, constraints, issues, metrics, and artifacts.
3. **Workflow Runtime** defines permitted workflows and controls task state
   transitions. Work is performed through a workflow.
4. **Verification Runtime** defines and records proof of completion. `Done`
   means the required verification policy passed.
5. **Memory Runtime** builds relevant, durable context for a task or agent.
   Agents should not scan an entire repository by default.
6. **Decision Runtime** records consequential choices, reasons, alternatives,
   and which decision is currently authoritative.
7. **Agent Runtime** registers agents and binds them to roles, capabilities,
   permissions, and assignments. An agent is not a prompt string.
8. **Workspace Runtime** projects the bounded world an agent needs: Mission,
   Goal, Task, Workflow, Role, related Knowledge and Constraints, pending
   Decisions, Verification status, and permitted next actions.

## 4. First-class objects

PKR v0.1 identifies fourteen stable core object kinds:

1. Mission
2. Goal
3. Task
4. Knowledge
5. Decision
6. Constraint
7. Workflow
8. Verification
9. Artifact
10. Metric
11. Agent
12. Role
13. Issue
14. Release

Packages may extend these objects but must not redefine their identity or core
semantics.

## 5. Project lifecycle

```text
Mission
  -> Goal
  -> Task
  -> Knowledge and Decision
  -> Implementation artifacts
  -> Verification
  -> Release
  -> Feedback
  -> Evolution
```

`Implementation`, `Feedback`, and `Evolution` are lifecycle concepts, not new
core object kinds. They are represented by workflows, tasks, artifacts,
metrics, issues, decisions, and events.

## 6. Domain packages

The core remains domain-neutral. Packages contribute domain schemas, policies,
workflows, and verification rules by extending core objects. Examples include
MMO concepts such as Player, Guild, Quest, Economy, and Inventory; FPS
concepts such as Weapon, Projectile, Animation, and Replication; and SaaS
concepts such as Customer, Subscription, Invoice, and Organization.

A package does not replace a core object or create a second lifecycle.

## 7. Agent contract

Before performing a mutating action, an agent must load:

1. the active Mission;
2. the current Goal and Task;
3. the applicable Workflow;
4. relevant Knowledge and Constraints;
5. its assigned Role and permissions;
6. the required Verification policy;
7. the applicable Decision basis.

An agent does not mutate the repository outside the runtime contract. The
runtime authorizes the action and records its outcome as artifacts and events.

## 8. Logical project layout

```text
my-project/
|-- .pkr/
|   |-- mission/
|   |-- goals/
|   |-- knowledge/
|   |-- decisions/
|   |-- workflows/
|   |-- verification/
|   |-- agents/
|   |-- roles/
|   |-- runtime/
|   |-- packages/
|   |-- memory/
|   |-- metrics/
|   `-- releases/
|-- src/
|-- tests/
`-- docs/
```

This is an illustrative filesystem projection, not a requirement that every
runtime implementation use file storage.

## 9. Roadmap

### Phase 0: Specification

Define the PKR Manifest, Object Model, lifecycle, Runtime API, Steward and
orchestration boundary, Workflow and Verification profile, Agent Session
protocol, Workspace and Memory contract, and Package evolution policy.

### Phase 1: Reference Runtime

Prove the specification with a minimal TypeScript implementation, a local
transactional event store, deterministic filesystem projections, and one
end-to-end Steward and LPS flow. The minimum surface includes `pkr init`,
`pkr status`, `pkr task`, `pkr dispatch`, `pkr callback`, `pkr verify`,
`pkr events`, and `pkr workspace`.

### Phase 2: Memory and Package Ecosystem

Enable persistent provenance-aware Memory, custom project Workflows, project
profiles, and independently developed namespaced Packages while keeping the
core Runtime stable.

### Phase 3: Governed Evolution and Compatibility

Connect heterogeneous Agent providers through one PKR Contract, automate
bounded improvement proposals, verify candidate behavior in sandboxes or
canaries, and provide compatibility, recovery, promotion, and rollback tooling.

## 10. v0.2 problem statement

The next specification must define the PKR Object Model (POM): common fields,
object-specific fields, relationships, lifecycles, events, and invariants. It
must answer at least:

- Which links are mandatory for a Task?
- What permits a Task to move from `Todo` to `Done`?
- Which verification results are required and which may be waived?
- What makes a Decision valid and when must one be created?
- How do packages extend objects without redefining the runtime?
- Which data is authoritative and which data is a derived Workspace or Memory
  projection?

## 11. v0.3 coordination problem statement

The v0.3 specification set must make the v0.2 model operable through one
human-to-Steward-to-orchestrator-to-Agent loop. It must answer at least:

- How can the Steward be the default human interface without becoming a root
  bypass?
- Which state belongs to PKR and which LPS state is only a projection?
- How are Agent identity, Assignment, Session, Lease, and provider thread kept
  distinct?
- How are custom state machines expressed without weakening POM lifecycles or
  executing arbitrary policy code?
- How does a callback become evidence without becoming automatic completion?
- How can Memory persist across sessions while remaining derived and
  provenance-bound?
- How can a project improve its own Workflows, prompts, adapters, and Packages
  without self-approving privilege or policy changes?
- Which exact golden path must the reference Runtime prove after restart?
