## 1. Purpose

This catalog freezes the bounded v0.4 record and action surface before the
reference Runtime is implemented. It identifies authority, mutation ownership,
revision and sequence rules, and protocol operations without adding a fifteenth
POM core object kind.

## 2. Authority classes

| Class | Meaning | Examples |
| --- | --- | --- |
| POM object | durable project intent and observed domain state | Task, Workflow, Agent, Verification |
| Runtime control record | durable execution control state owned by the Runtime | WorkflowRun, Assignment, AgentSession, Lease, PackageInstallation |
| Protocol record | attributable request, fact, or result; not mutable state | AgentMessage, CoordinationCommand, CoordinationEvent, CoordinationResult |
| Derived projection | rebuildable read model; never mutation authority | Workspace, MemoryProjection |
| Derived index entry | persisted retrieval aid with authoritative source links | MemoryEntry |
| External locator | provider or tool address with no PKR authority | provider thread ID, repository path, CI run URL |

POM objects and committed Runtime control records are authoritative within their
declared responsibility. Protocol records are immutable evidence of interaction.
Derived records never override a POM object or Runtime control record.

## 3. Common rules

- Every record belongs to exactly one `projectId`.
- Mutable control records have a positive `revision` and use optimistic
  concurrency.
- Every record or projection is bound to a `projectSequence`.
- Runtime-owned fields cannot be replaced by client input after commit.
- Every accepted mutation emits at least one immutable event in the same commit.
- Every command uses actor, Role, Workflow, scope, decision basis, idempotency,
  expected revision, and UTC issue time.
- Extensions are namespaced and cannot weaken core validation.
- Control records are logically deleted through terminal lifecycle states; they
  are not hard-deleted after emitting events.

## 4. Record catalog

### 4.1 WorkflowRun

- Authority class: Runtime control record.
- Identity: immutable `runId`.
- Client intent: exact Workflow and scope binding at creation; requested edge.
- Runtime-owned mutable state: local state, active/completed steps, pending
  gates, revision, project sequence, timestamps.
- Lifecycle: Workflow-defined; creation starts at exact `initialState`.
- Required references: Workflow ID/revision and Task or governance scope.
- Events: `pkr.workflowRun.started`, `pkr.workflowRun.transitioned`,
  `pkr.verification.attemptRecorded`.

### 4.2 CapabilityStatement

- Authority class: immutable protocol evidence with expiry.
- Identity: immutable `capabilityStatementId`.
- Owner: authenticated Agent adapter.
- Content: Agent ID, adapter ID/version, protocol versions, declared
  capabilities, limits, isolation properties, issued and expiry times.
- Rule: eligibility input only; effective permission is the intersection of
  Agent, Role, Workflow, Assignment, Lease, and host policy.
- Events: `pkr.agent.capabilitiesDeclared` when accepted by the Runtime.

### 4.3 Assignment

- Authority class: Runtime control record.
- Identity: immutable `assignmentId` plus immutable idempotency key.
- Client intent: bounded objective, Task/Workflow revisions, Role, allowed and
  forbidden scope, acceptance, required capabilities, expected Artifacts, and
  callback contract.
- Runtime-owned mutable state: lifecycle, revision, project sequence,
  disposition, timestamps.
- Lifecycle: `prepared -> offered -> accepted -> running <-> blocked ->
  submitted -> closed`, with `rejected`, `cancelled`, `expired`, and `failed`
  terminal alternatives.
- Events: `pkr.assignment.offered`, `accepted`, `started`, `blocked`,
  `submitted`, `closed`, `cancelled`, `expired`, and `failed`.

### 4.4 AgentSession

- Authority class: Runtime control record.
- Identity: immutable `sessionId`; provider locator is not identity.
- Client intent: Agent, provider locator, adapter, protocol version, capability
  statement, and eligible Assignment links.
- Runtime-owned mutable state: lifecycle, Assignment set, heartbeat, expiry,
  revision, project sequence, timestamps.
- Lifecycle: `opening -> active <-> suspended -> closing -> closed`, with
  `failed` and `expired` terminal alternatives.
- Events: `pkr.agentSession.opened`, `activated`, `suspended`, `closing`,
  `closed`, `failed`, and `expired`.

### 4.5 Lease

- Authority class: Runtime control record.
- Identity: immutable `leaseId`.
- Client intent: Assignment, Session, Agent, requested scope, mode, and duration.
- Runtime-owned mutable state: lifecycle, expiry, heartbeat observation,
  revision, project sequence, timestamps.
- Lifecycle: `active -> renewed -> released|revoked|expired` where renewal
  remains semantically active.
- Invariant: one active exclusive holder per Assignment scope.
- Events: `pkr.lease.acquired`, `renewed`, `released`, `revoked`, and `expired`.

### 4.6 AgentMessage

- Authority class: immutable protocol record.
- Identity: immutable idempotent `messageId`.
- Owner: authenticated sender; routing and acceptance are Runtime-owned.
- Required correlation: Project, sender, recipient, Assignment where applicable,
  correlation ID, causation ID when applicable, source project sequence, and
  issue time.
- Message families: Assignment response, execution progress/heartbeat/blocker,
  callback, handoff, cancellation, Session closure, Lease operation, and error.
- Rule: message content does not become project truth without the applicable
  governed mutation.

### 4.7 WorkspaceRequest

- Authority class: immutable read request.
- Identity: immutable idempotent `requestId`.
- Owner: authenticated requesting principal.
- Content: Role, Task/Assignment/governance scope, projection profile, maximum
  size, sequence selector, requested time.
- Rule: cannot widen current read permission.

### 4.8 Workspace

- Authority class: immutable derived projection.
- Identity: immutable `workspaceId`.
- Owner: Runtime projection builder.
- Content: principal, Role, project sequence, expiry, mandatory execution
  context, permitted and forbidden actions, Memory entry references, and
  truncation/redaction notices.
- Rule: commands use authoritative revisions; Workspace itself is never a
  mutation source.

### 4.9 MemoryEntry

- Authority class: derived index entry.
- Identity: immutable `memoryId` for one derivation version.
- Owner: Runtime Memory projector.
- Content: class, summary, complete source references, event ranges, derivation
  method/version, confidence, retention, visibility, validity, and invalidation.
- Rule: `derived` is always true; confidence affects rank only.
- Events: source promotion is a separate POM command, not a Memory mutation.

### 4.10 MemoryProjection

- Authority class: immutable derived projection.
- Identity: immutable `projectionId`.
- Owner: Runtime Memory projector.
- Content: principal, project sequence, selected entry IDs, visibility profile,
  expiry, and status.
- Rule: invalid or orphaned entries must be excluded or explicitly labeled.

### 4.11 PackageManifest

- Authority class: immutable package protocol record.
- Identity: immutable package ID, semantic version, and content digest tuple.
- Owner: publisher identity.
- Content: compatibility, dependencies, conflicts, contributions, requested
  capabilities, migration, rollback, license, and distribution metadata.
- Rule: the same ID/version cannot identify different content.

### 4.12 PackageInstallation

- Authority class: Runtime control record.
- Identity: immutable `installationId`.
- Client intent: Package identity, resolved dependency set, approved capability
  ceiling, contributions, Decision, and exact Workflow revision.
- Runtime-owned mutable state: lifecycle, health, migration and rollback status,
  revision, project sequence, timestamps.
- Lifecycle: `proposed -> resolving -> staged -> active -> suspended ->
  uninstalled`, with `failed` and `superseded` alternatives.
- Events: every lifecycle edge plus migration and rollback results.

## 5. Action catalog

| Action | Target | Create revision | Required semantic guard |
| --- | --- | --- | --- |
| `startWorkflowRun` | WorkflowRun | 0 | exact active Workflow and initial state |
| `transitionWorkflowRun` | WorkflowRun | current | legal edge, Role, guards, gates, Constraints |
| `recordVerificationAttempt` | WorkflowRun | current | immutable attempt and applicable target revision |
| `offerAssignment` | Assignment | 0 | committed Task/Workflow and bounded capability request |
| `respondAssignment` | Assignment | current | compatible Agent and active Role |
| `openAgentSession` | AgentSession | 0 | active Agent and accepted capability statement |
| `closeAgentSession` | AgentSession | current | legal closure reason and Assignment disposition |
| `acquireLease` | Lease | 0 | accepted Assignment and no conflicting holder |
| `renewLease` | Lease | current | current active holder and unchanged scope |
| `releaseLease` | Lease | current | current holder or authorized orchestrator |
| `revokeLease` | Lease | current | authorized policy or owner action |
| `submitAgentMessage` | AgentMessage | 0 | authenticated sender and valid correlation |
| `proposePackageInstall` | PackageInstallation | 0 | immutable manifest identity and Decision basis |
| `resolvePackageInstall` | PackageInstallation | current | deterministic dependencies and conflicts |
| `stagePackageInstall` | PackageInstallation | current | schema, capability, migration, and rollback checks |
| `activatePackageInstall` | PackageInstallation | current | staged version and required Verification |
| `suspendPackageInstall` | PackageInstallation | current | authorized active installation |
| `uninstallPackage` | PackageInstallation | current | uninstall policy and preserved history |
| `rollbackPackage` | PackageInstallation | current | declared compatible prior target |

## 6. Read catalog

| Read | Consistency |
| --- | --- |
| `getWorkflowRun` / `listWorkflowRuns` | exact revision or one project snapshot sequence |
| `getControlRecord` / `listControlRecords` | exact revision or one project snapshot sequence |
| `getWorkspace` | newly built immutable projection at a declared sequence |
| `getMemoryProjection` | immutable visible selection at a declared sequence |
| `getPackageInstallation` / `listPackageInstallations` | exact revision or one project snapshot sequence |

Unsupported historical reads fail explicitly and never return latest state as a
substitute.

## 7. Stable v0.4 errors

| Code | Meaning |
| --- | --- |
| `PKR-COORD-001` | unsupported coordination record, action, or API version |
| `PKR-COORD-002` | structural schema validation failed |
| `PKR-COORD-003` | target revision or project sequence is stale |
| `PKR-COORD-004` | lifecycle edge or Workflow guard failed |
| `PKR-COORD-005` | capability or effective permission is insufficient |
| `PKR-COORD-006` | Assignment, Session, Lease, or message correlation is invalid |
| `PKR-COORD-007` | exclusive Lease conflicts with an active holder |
| `PKR-COORD-008` | callback is late, unowned, unverified, or outside scope |
| `PKR-COORD-009` | Workspace or Memory source is stale, invisible, or orphaned |
| `PKR-COORD-010` | Package identity, dependency, capability, or compatibility failed |
| `PKR-COORD-011` | package staging, activation, migration, or rollback failed atomically |
| `PKR-COORD-012` | idempotency key was reused with different content |

## 8. Deferred identities

v0.4 does not add Scheduler, Conversation, Plan, Prompt, ProviderThread,
Organization, Secret, Credential, Model, or EvolutionProposal as POM core object
kinds. Scheduling remains Workflow and orchestration behavior. Conversation and
prompt material are Artifacts or Knowledge when governed. Evolution proposals
are represented through Issue, Decision, Task, Artifact, Verification, Release,
Package, and event records.

## 9. Conformance boundary

Structural conformance proves record and action shapes. Semantic conformance
proves cross-record lifecycle, authority, idempotency, Lease, callback,
projection, Memory, and Package invariants. Neither structural fixtures nor the
v0.4 in-memory semantic harness is a reference Runtime or persistence proof.
