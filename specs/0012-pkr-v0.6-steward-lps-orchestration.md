# PKR v0.6 Steward and LPS Orchestration

- Status: Implemented milestone
- Index: [PKR specification index](README.md)
- Scope: Governed intake, Agent assignment, orchestration, and recovery
- Depends on: [PKR v0.5 Reference Runtime](0011-pkr-v0.5-reference-runtime.md)
- Successor: [PKR v0.7 Persistent Intelligence](0013-pkr-v0.7-persistent-intelligence.md)
- Publication: Not independently released

## 1. Purpose

v0.6 connects a human-facing Project Steward and an LPS orchestration adapter
to the Runtime without creating a second source of project truth. It defines
the minimum governed path from a human request to independently verified Agent
work.

## 2. Steward contract

The Steward prepares deterministic proposals before mutation. A material
proposal MUST remain pending until the Manifest owner explicitly approves it.
An accepted Decision MUST exist before the corresponding material Goal or Task
mutation is committed. Conversation silence, a recommendation, or model output
is never approval.

## 3. Orchestration contract

LPS MUST derive its live board from Runtime Assignment, AgentSession, Lease,
Task, Verification, and event sequence state. It MAY cache or render a board,
but that projection MUST NOT become mutation authority.

The governed execution path covers:

- capability-aware Agent registration and Assignment;
- Session and Lease creation, heartbeat, expiry, and cancellation;
- a bounded Workspace request and structured callback;
- duplicate dispatch and callback replay;
- late-callback rejection and timeout recovery;
- Session and WorkflowRun closure after verified Task completion.

## 4. Provider boundary

The reference process Adapter proves bounded stdin/stdout integration, timeout,
capability declaration, and callback handling. Provider thread identifiers and
reports remain non-authoritative. Fixture or local-process success does not
prove a live model service, external Agent host, or production integration.

## 5. Recovery invariants

- Restart MUST rebuild an equivalent orchestration view from Runtime state.
- Lease expiry or cancellation MUST prevent new governed effects.
- Re-running a closed lane MUST NOT create duplicate events or live workers.
- A participant callback MUST NOT bypass Repository Verification or Runtime
  acceptance.

## 6. Compatibility role

v0.6 is an implemented and verified development milestone. Its provider-neutral
identity, Lease, report, Verification, and derived-board boundaries continue in
the [LPS integration contract](../docs/integrations/lps.md) and the accepted v1
authority model.

## 7. Non-goals

v0.6 does not define a graphical UI, select a model Provider automatically,
support every Provider family, or authorize self-directed backlog expansion.
