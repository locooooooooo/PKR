# PKR v1.2 Supervisor and Runner

- Status: Stable source release with experimental Supervisor surface
- Index: [PKR specification index](README.md)
- Scope: Explicit one-Task reconciliation, effect fencing, and Owner attention
- Depends on: [PKR v1.1 Evidence Integrity and Sharing](0017-pkr-v1.1-evidence-integrity.md)
- Release: [`v1.2.0`](https://github.com/locooooooooo/PKR/releases/tag/v1.2.0)

## 1. Purpose

v1.2 adds an optional, explicitly started, Provider-neutral Supervisor/Runner
that advances one governed Task by at most one deterministic reconciliation
action at a time. It preserves the accepted v1 authority and acceptance model.

## 2. Configuration contract

`pkr.supervisor/v1` binds one existing Task, active Agent, explicit local
Provider Adapter, independent verifier, Verification plan, and required
capabilities. PKR MUST NOT select a Provider automatically or infer identity,
permission, approval, or acceptance from Skill metadata.

## 3. Reconciliation contract

- `pkr supervise --once` performs at most one action and returns a structured
  `pkr.supervisor-result/v1` result.
- `pkr supervise --watch` repeats the same composable step at an explicit
  interval.
- The loop stops when the Task is terminal, blocked, requires clarification,
  or needs Owner attention.
- Lease heartbeat, expiry, and one governed recovery dispatch reuse existing
  Runtime and LPS transitions.

## 4. Effect fencing

Provider and Verification effects are reserved and journaled in SQLite before
external execution. A recorded successful effect replays after restart. An
ambiguous pending effect MUST stop for attention instead of running twice.
Concurrent attempts MUST NOT create duplicate live Provider or Verification
effects.

## 5. Authority invariants

Provider reports remain non-authoritative. Independent Repository Verification
remains the only evidence path to acceptance, and Runtime acceptance remains a
separate transition. The Supervisor coordinates existing governed operations;
it does not own project truth, approve protected decisions, or mark work done
from a participant narrative.

## 6. Compatibility role

v1.2 is a backward-compatible stable source release. The `supervise` CLI route,
Supervisor exports, configuration/result contracts, and narrow Verification
effect reservation API remain experimental under the
[v1 stable contract](../docs/release/v1-stable-contract.md). No v1.1 stable
surface is removed or reclassified.

## 7. Operational boundary

The Supervisor is explicit local liveness infrastructure, not an always-on
hosted service, Provider marketplace, OS sandbox, or production scheduler.
Fake and local Provider fixtures prove protocol behavior only. See the
[v1.2 release specification](../docs/release/v1.2.0.md) and
[Runtime guide](../docs/runtime-guide.md).
