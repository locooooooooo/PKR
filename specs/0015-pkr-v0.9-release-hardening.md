# PKR v0.9 Compatibility and Release Hardening

- Status: Integrated milestone
- Index: [PKR specification index](README.md)
- Scope: Compatibility, recovery, security, performance, and release evidence
- Depends on: [PKR v0.8 Governed Evolution](0014-pkr-v0.8-governed-evolution.md)
- Successor: [PKR v1.0 Stable Contract](0016-pkr-v1.0-stable-contract.md)
- Publication: Folded into v1.0; not independently released

## 1. Purpose

v0.9 is the release-hardening bridge between the verified feature milestones
and the first stable contract. It replaces single-machine success claims with
explicit compatibility, recovery, security, packaging, and publication gates.

## 2. Compatibility contract

- Provider-specific fields remain namespaced and non-authoritative.
- Different Agent integrations MUST preserve the same Task, report,
  Verification, and acceptance semantics.
- Adapter fixture conformance MUST be reported separately from live Provider
  evidence.
- Supported schema, record, event, CLI, and TypeScript surfaces MUST be listed
  explicitly before a stable compatibility claim.

## 3. Recovery and integrity contract

The Runtime MUST define supported snapshot, restore, retention, migration, and
downgrade behavior. Interrupted Sessions, expired Leases, reassignment, and
external effects MUST fail closed without duplicate live work or silent
acceptance. Compaction MUST preserve every record and evidence claim required
by the declared retention policy.

## 4. Security and operational contract

Release readiness requires:

- a trusted-host threat model and explicit residual risks;
- bounded, redacted diagnostics and sensitive-data scanning;
- dependency and license provenance;
- deterministic benchmark and restart-soak procedures;
- source and packed-artifact installation checks;
- Windows and Ubuntu CI for the exact candidate;
- a reviewed public-tree allowlist and clean release identity.

Local green tests alone do not prove a live Provider, OS sandbox, production
security, SLA, workload, or adoption.

## 5. Publication contract

GitHub source publication, Git tag creation, GitHub Release creation, and npm
publication are separate transactions. Apache-2.0, NOTICE, third-party notices,
package metadata, and public documentation MUST agree. Remaining blockers MUST
be explicit owner Decisions or Issues rather than hidden TODOs.

## 6. Compatibility role

v0.9 was not tagged or released independently. Its outputs became the v1
contract inventory, gate register, migration and recovery rules, threat model,
release checks, and evidence classification used by v1.0.
